# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Single-user personal diary running entirely on Cloudflare Workers. React Router v8 framework mode (SSR) on the Worker, with D1 (SQL), R2 (attachments), KV (sessions), Vectorize (semantic search), and Workers AI. Storage is server-side with at-rest encryption (not E2EE) — the server reads plaintext so it can run full-text search and AI. UI copy is Chinese.

## Commands

```bash
npm run dev              # wrangler dev (Miniflare): http://localhost:5173
npm run typecheck        # wrangler types && react-router typegen && tsc -b  (run after wrangler.jsonc changes)
npm test                 # vitest run (real workerd runtime)
npm test -- --run tests/backup.test.ts   # single test file
npm run test:watch       # vitest watch
npm run test:coverage    # coverage; CI gate is 80% on all metrics (vitest.config.ts)
npm run lint             # biome check .   (Biome, not ESLint/Prettier)
npm run format           # biome format --write .
npm run db:generate      # drizzle-kit generate — SQL migrations from schema.ts
npm run db:migrate:local # wrangler d1 migrations apply diary-db --local
npm run deploy           # build && wrangler deploy  (needs real resource IDs + secrets first)
```

Migrations are **generated** by drizzle-kit but **applied** by `wrangler d1 migrations apply` (drizzle.config.ts has no DB credentials). Tests apply migrations themselves (see below), so no manual migrate step is needed before running them.

## Architecture

**Worker entry — `workers/app.ts`.** The `fetch` handler creates a React Router `RouterContextProvider`, injects `ctx.waitUntil` into it via `waitUntilContext` (`app/lib/request-context.ts`), then delegates to the RR request handler. The `scheduled` handler runs the daily backup. Routes reach background work through that injected context — see the AI pipeline below.

**Bindings & `Env`.** Bindings are `DB` (D1), `BLOBS` (R2), `SESSIONS` (KV), `VEC` (Vectorize), `AI` (Workers AI). The `Env` type is **generated** by `wrangler types` into `worker-configuration.d.ts` — never hand-write it, and it's gitignored (regenerated by `postinstall`/`typecheck`). In routes, get the env via `import { env } from "cloudflare:workers"` (a module global), not from loader/action arguments.

**Data layer (`app/lib/db/`).** Drizzle ORM; schema in `schema.ts`. Access is via small repository-style modules per aggregate (`entries.ts`, `tags.ts`, `calendar.ts`), each taking a `Db` from `getDb(env.DB)`. Timestamps are `timestamp_ms` Date columns. Keyword search uses a D1 FTS5 virtual table (trigram tokenizer for CJK) kept in sync by triggers in the migrations.

**Auth (`app/lib/auth/`).** One password gates everything: PBKDF2 hash compared against `APP_PASSWORD_HASH`, session stored in KV via RR `createSessionStorage` (cookie holds only an opaque id; HttpOnly/Secure/SameSite). `requireSession(request, env)` guards loaders/actions. The login action is rate-limited per client IP (`rate-limit.ts`, KV-backed, 10 fails / 15 min). `safeRedirect` blocks open-redirects on `redirectTo`.

**Attachments (`app/lib/storage/`).** Bytes are AES-256-GCM encrypted at the app layer with `ATTACH_ENCRYPTION_KEY` before being written to R2. The read route (`routes/attachment.tsx`) is session-guarded, decrypts, and streams. Upload validates type/size at the boundary.

**AI pipeline (`app/lib/ai/`).** After a save, enrichment (summary, sentiment, embedding → Vectorize upsert) runs **off the request critical path** via `context.get(waitUntilContext)(runAiPipeline(env, entry))` — never `await`ed in the action. The pipeline never fails a save: provider errors are captured as `skippedReasons`. Providers default to `disabled`: text = `disabled|workers-ai|anthropic|openai`, embedding = `disabled|workers-ai|openai`. Anthropic and OpenAI-compatible endpoints are called via raw `fetch`; Workers AI via `env.AI.run`. Remote calls happen only when a provider is selected **and** its API key is resolved.

Config resolution: the pipeline calls `loadAiConfig(env)` (async), which layers **web-managed settings over env**. `getAiConfig(env)` reads env vars/secrets (the fallback); `loadAiSettings(env)` (`app/lib/settings/ai-settings.ts`) reads web settings from KV. When web settings are present and enabled, they drive the **text/chat provider** (protocol `anthropic|openai`, base URL, model, key); embeddings stay on env. The web API key is **AES-256-GCM encrypted at rest** (KV, `ATTACH_ENCRYPTION_KEY`) and never sent to the browser — the loader returns only a `hasApiKey` view. The `/settings` route saves these and offers a live "test connection" (`test-connection.ts`) that pings the provider. `generateText`/`embedText` read the key from the resolved config, falling back to `env.*_API_KEY`.

**Search (`app/lib/search/`).** `fts.ts` (D1 FTS5 keyword) and Vectorize semantic results are merged/deduped in `hybrid.ts`.

**Backup (`app/lib/backup/export.ts`).** The daily cron (`triggers.crons` in wrangler.jsonc) exports the whole diary as JSON + Markdown to R2 under `backups/<date>/diary-<timestamp>.{json,md}`. `runScheduledBackup` logs failures and rethrows so a failed cron run is visible.

Path alias: `~/*` → `app/*` (mirrored in both tsconfig and vitest.config.ts).

## Testing

Tests run inside the real workerd runtime via `@cloudflare/vitest-pool-workers`, so D1/R2/KV bindings behave as in production. `tests/apply-migrations.ts` applies the D1 migrations per test-worker; each test gets isolated, rolled-back storage. Import the test env with `import { env } from "cloudflare:test"`. For provider/`fetch` logic, stub globals with `vi.stubGlobal("fetch", ...)`. **Vectorize has no local emulation** — semantic-search paths can't be exercised in local tests or `wrangler dev` (they degrade to FTS).

## Deployment notes

`wrangler.jsonc` resource IDs (`database_id`, KV `id`) are **placeholders for local dev**. Real deploy requires creating the resources (`wrangler d1 create` / `kv namespace create` / `r2 bucket create` / `vectorize create --dimensions=1024 --metric=cosine`), replacing the IDs, and setting secrets with `wrangler secret put`: `SESSION_SECRET`, `APP_PASSWORD_HASH`, `ATTACH_ENCRYPTION_KEY` (+ `ANTHROPIC_API_KEY`/`OPENAI_API_KEY` if AI is enabled). Local secrets live in `.dev.vars` (gitignored); only `.dev.vars.example` is committed — it documents how to generate each value.
