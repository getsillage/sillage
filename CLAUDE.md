# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project

Single-user personal diary running entirely on Cloudflare Workers. React Router v8 framework mode (SSR) on the Worker, with D1 (SQL), R2 (attachments), and KV (sessions). Storage is server-side with at-rest encryption (not E2EE) — the server reads plaintext so it can run full-text search and AI summaries. UI copy is Chinese.

## Commands

```bash
npm run dev              # wrangler dev (Miniflare): http://localhost:5173
npm run typecheck        # wrangler types && react-router typegen && tsc -b  (run after wrangler.jsonc changes)
npm test                 # vitest run (real workerd runtime)
npm test -- --run tests/backup.test.ts   # single test file
npm run test:watch       # vitest watch
npm run test:coverage    # coverage; CI gate is 80% on all metrics (vitest.config.ts)
npm run lint             # biome check .   (Biome, not ESLint/Prettier)
npm run format           # biome format --write .
npm run db:generate      # drizzle-kit generate — SQL migrations from schema.ts
npm run db:migrate:local # wrangler d1 migrations apply diary-db --local
npm run deploy           # build && wrangler deploy  (needs real resource IDs + secrets first)
```

Migrations are **generated** by drizzle-kit but **applied** by `wrangler d1 migrations apply` (drizzle.config.ts has no DB credentials). Tests apply migrations themselves (see below), so no manual migrate step is needed before running them.

## Architecture

**Worker entry — `workers/app.ts`.** The `fetch` handler creates a React Router `RouterContextProvider`, injects `ctx.waitUntil` into it via `waitUntilContext` (`app/lib/request-context.ts`), then delegates to the RR request handler. The `scheduled` handler runs the daily backup. Routes reach background work through that injected context — see the AI pipeline below.

**Bindings & `Env`.** Bindings are `DB` (D1), `BLOBS` (R2), and `SESSIONS` (KV). The `Env` type is **generated** by `wrangler types` into `worker-configuration.d.ts` — never hand-write it, and it's gitignored (regenerated by `postinstall`/`typecheck`). In routes, get the env via `import { env } from "cloudflare:workers"` (a module global), not from loader/action arguments.

**Data layer (`app/lib/db/`).** Drizzle ORM; schema in `schema.ts`. Access is via small repository-style modules per aggregate (`entries.ts`, `tags.ts`, `calendar.ts`), each taking a `Db` from `getDb(env.DB)`. Timestamps are `timestamp_ms` Date columns. Keyword search uses a D1 FTS5 virtual table (trigram tokenizer for CJK) kept in sync by triggers in the migrations.

The schema is **sync-/multi-client-oriented**: ids are time-sortable **UUIDv7** (`id.ts`, mint client-side, usable as a cursor); every aggregate carries `updatedAt` (indexed) + a soft-delete `deletedAt` tombstone; `entries` also has an optimistic-concurrency `version`, an `utcOffsetMinutes`, and a forward-compatible JSON `metadata` bag. `deleteEntry` **soft-deletes** (sets the tombstone, preserves tag links/attachments for undo; `restoreEntry`/`purgeEntry` complete the lifecycle); reads filter `deletedAt IS NULL`, the FTS triggers drop/re-add rows on the tombstone, and `updateEntry` returns `{status: updated|missing|conflict}` and never wipes `metadata`/offset that the input omits. **Machine-derived AI output lives in the `entry_ai` side table** (1:1), read back via a left join in `composeEntries`/`fts.ts`/backup — so regenerating a summary never bumps `entries.updatedAt` or churns FTS. `sync.ts` (`getChangesSince`) is the delta-read model behind the sync API.

**Auth (`app/lib/auth/`).** One password gates everything: PBKDF2 hash compared against `APP_PASSWORD_HASH`, session stored in KV via RR `createSessionStorage` (cookie holds only an opaque id; HttpOnly/Secure/SameSite). `requireSession(request, env)` guards loaders/actions. The login action is rate-limited per client IP (`rate-limit.ts`, KV-backed, 10 fails / 15 min). `safeRedirect` blocks open-redirects on `redirectTo`.

**Attachments (`app/lib/storage/`).** Bytes are AES-256-GCM encrypted at the app layer with `ATTACH_ENCRYPTION_KEY` before being written to R2. The read route (`routes/attachment.tsx`) is session-guarded, decrypts, and streams. Upload validates type/size at the boundary.

**AI pipeline (`app/lib/ai/`).** After a save, summary generation runs **off the request critical path** via `context.get(waitUntilContext)(runAiPipeline(env, entry))` — never `await`ed in the action. The pipeline never fails a save: provider errors are captured as `skippedReasons`. Providers default to `disabled`; enabled web profiles use `anthropic|openai`. Anthropic and OpenAI-compatible endpoints are called via raw `fetch`. Remote calls happen only when a provider is selected **and** its API key is resolved. The result is **upserted into `entry_ai`** (with the originating `model`), never written back to `entries`, keeping the sync feed and FTS index quiet for a purely derived change.

Config resolution: the pipeline calls `loadAiConfig(env)` (async), which reads **web-managed settings** from KV. When web settings are present and enabled, they drive the summary provider (protocol `anthropic|openai`, base URL, model, key). The web API key is **AES-256-GCM encrypted at rest** (KV, `ATTACH_ENCRYPTION_KEY`) and never sent to the browser — the loader returns only a `hasApiKey` view. The `/settings` route saves multiple profiles and offers model listing plus a live "test connection" (`test-connection.ts`) that pings the provider.

**Search (`app/lib/search/`).** `fts.ts` implements D1 FTS5 keyword search.

**Sync API (`routes/api.sync.tsx` + `app/lib/api/serialize.ts`).** `GET /api/sync?since=<iso|ms>` (session-guarded) returns everything changed after the cursor — including soft-deleted tombstones so clients can mirror removals — plus a `cursor` to pass back. Rows are mapped through the `serialize.ts` DTO layer (ISO 8601 timestamps, parsed `metadata`, R2 keys dropped, attachment `url`) so the internal schema can evolve without breaking non-web clients (e.g. a mobile app).

**Backup (`app/lib/backup/export.ts`).** The daily cron (`triggers.crons` in wrangler.jsonc) exports the whole diary as JSON + Markdown to R2 under `backups/<date>/diary-<timestamp>.{json,md}`. `runScheduledBackup` logs failures and rethrows so a failed cron run is visible.

Path alias: `~/*` → `app/*` (mirrored in both tsconfig and vitest.config.ts).

## Testing

Tests run inside the real workerd runtime via `@cloudflare/vitest-pool-workers`, so D1/R2/KV bindings behave as in production. `tests/apply-migrations.ts` applies the D1 migrations per test-worker; each test gets isolated, rolled-back storage. Import the test env with `import { env } from "cloudflare:test"`. For provider/`fetch` logic, stub globals with `vi.stubGlobal("fetch", ...)`.

## Deployment notes

`wrangler.jsonc` resource IDs (`database_id`, KV `id`) are **placeholders for local dev**. Real deploy requires creating the resources (`wrangler d1 create` / `kv namespace create` / `r2 bucket create`), replacing the IDs, and setting secrets with `wrangler secret put`: `SESSION_SECRET`, `APP_PASSWORD_HASH`, `ATTACH_ENCRYPTION_KEY`. AI provider/model/API key settings are managed in the web `/settings` page. Local secrets live in `.dev.vars` (gitignored); only `.dev.vars.example` is committed — it documents how to generate each value.
