# Support

Sillage is a self-hosted open-source project without an official hosted service or a support SLA. Maintainers and community members provide help on a best-effort basis.

## Before Asking

1. Read the [Documentation Hub](docs/README.md), especially the [Deployment Guide](docs/user/deployment.md) and [Data, Backup, and Recovery](docs/user/data.md).
2. Confirm that you are using the latest release or identify the exact commit you built.
3. Search existing [GitHub Issues](https://github.com/getsillage/sillage/issues) for the same problem or question.
4. Remove or replace all records, attachment names, hostnames, credentials, API keys, tokens, database contents, and other sensitive data.

## Where to Ask

- Use the [Bug Report form](https://github.com/getsillage/sillage/issues/new?template=bug_report.yml) for a reproducible defect.
- Use the [Feature Request form](https://github.com/getsillage/sillage/issues/new?template=feature_request.yml) for an improvement within Sillage's private, single-user product scope.
- Use the [Support Question form](https://github.com/getsillage/sillage/issues/new?template=question.yml) for setup, configuration, or usage questions that are not reproducible defects.
- Report vulnerabilities only through [GitHub Private Vulnerability Reporting](https://github.com/getsillage/sillage/security/advisories/new), following the [Security Policy](SECURITY.md).
- Report conduct concerns privately as described in the [Code of Conduct](CODE_OF_CONDUCT.md).

## Support Boundaries

- Public support is limited to sanitized information. Maintainers cannot inspect private instances, records, databases, attachments, secrets, or backups.
- Sillage has no built-in password recovery, and data recovery depends on a valid backup. Follow the documented backup and restore procedures before upgrades or migrations.
- Questions about custom reverse proxies, tunnels, AI providers, operating systems, or networks may require support from those upstream projects.
- Older releases are not guaranteed to receive separate fixes or troubleshooting.
- Do not use public Issues for security vulnerabilities, conduct reports, or sensitive incident details.
