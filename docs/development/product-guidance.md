# Sillage 产品指导文件

## 文档用途

本文用于约束 Sillage 的功能设计、界面文案、交互取舍和 AI 行为。后续设计以“用户容易理解”为优先，不使用偏诗意或抽象的表达。

本文不讨论底层实现、数据库结构、接口设计、存储方案或迁移步骤。

## 产品定位

Sillage 是一个单用户个人记录空间，用来保存每天发生的事、想法和感受。AI 的作用是基于这些记录生成总结，并支持用户按历史内容提问。

自托管实例首次访问时创建唯一账号；初始化完成后默认需要登录使用。产品不提供公开试用模式、匿名免登录模式或第二个账号注册入口。

一句话说明：

> Sillage 是一个个人记录空间，用来保存记录、查看历史、根据记录提问。

## 客户端形态

Sillage 当前包含 Web 客户端和原生 Android 客户端：

- Web 客户端覆盖完整主流程：初始化、登录、写记录、查看全部记录、问答、设置、附件和 AI 总结。
- Android 客户端覆盖手机核心流程：在线/离线模式、服务器地址配置、初始化/登录、记录列表/日历/搜索、新建/编辑/删除、收藏/归档、附件插入、AI 设置、记录总结、Ask、导入导出和手动同步。

Android 客户端仍必须遵守单人私密空间、中文 UI 使用“记录”、AI 回答基于来源记录这些产品原则。当前不做后台自动同步、推送通知或多人协作能力。

Android 移动端交互遵循以下已落地约定：

- 原生返回键与页面返回按钮保持同一语义。服务器连接、记录详情和记录编辑会回到实际来源页；从问答来源记录进入详情并编辑时，返回顺序仍是「编辑 → 详情 → 问答」。
- 新建或编辑记录时，只要正文或日期发生变化，离开编辑页前必须确认是否放弃；保存后以已保存内容作为新的草稿基线。
- 登录 / 初始化表单与记录编辑页使用可滚动布局、软键盘内边距与窗口 `adjustResize`，目标是让键盘弹出后仍能访问底部操作。
- 附件上传期间，记录编辑器暂停保存、返回和其他会改变当前记录的操作；上传结果只允许写回发起上传的同一次编辑会话，晚到结果不能插入后来打开的草稿。
- Markdown 中普通 `http/https` 外链交给系统浏览器；相对或绝对同源的 `/file/attachments/...` 必须由 App 携带当前认证下载到缓存，再经 `FileProvider` 以只读 URI 交给系统查看器。危险协议、跨目录路径和非法附件地址不执行。
- 打开问答来源时，返回的记录详情只在用户仍处于原问答页面、原会话与同一次页面会话时生效；用户已离开或切换会话后到达的旧响应不得把页面带回过期来源。
- 问答生成期间不允许切换会话或新建会话；流回调与最终刷新必须同时匹配原会话、页面会话、运行模式和请求 ID。列表分页同一游标只允许一个请求，刷新或游标变化后到达的旧分页结果不得写回。

上述 Android 交互已经代码落地，编辑器会话、问答流 / 来源、分页和附件地址分类均有状态或纯逻辑单测，`adjustResize` 与 `FileProvider` 也已在 debug 构建和 lint 中确认；键盘布局、上传中系统返回、延迟网络，以及第三方 PDF / 图片查看器的 chooser 交接仍需在模拟器或真机完成最终验收。

## 产品气质

Sillage 应该是：

- 私密的，而不是社交化的。
- 清楚的，而不是抽象的。
- 具体的，而不是空泛的。
- 易懂的，而不是依赖隐喻的。
- 适合长期使用的，而不是只为短期新鲜感设计的。
- 有依据的，而不是让 AI 凭空下结论。

Sillage 不应该是：

- 一个复杂的后台管理系统。
- 一个情绪打卡工具。
- 一个心理诊断工具。
- 一个效率复盘工具。
- 一个由 AI 主导表达的写作应用。
- 一个依赖诗意文案才能理解功能的产品。

## 核心原则

1. 记录入口要直接，让用户知道可以写什么。
2. 产品只区分「记录、历史、问答、总结」，不再区分记录种类。
3. 写作始终优先于整理，用户表达优先于 AI 输出。
4. AI 只能基于已有记录生成总结或回答问题。
5. 所有 AI 总结和回答都应能回到原始记录。
6. 产品文案要通俗、明确、具体。
7. 界面要安静、易读，但不能牺牲可用性。

## 核心模型

Sillage 由四个核心概念组成：

| 英文 | 中文 | 产品含义 |
| --- | --- | --- |
| Record | 记录 | 用户写下的一件事、一个想法、一种感受、一段文字或一张照片 |
| History | 历史 | 按时间查看过去的记录 |
| Ask | 问答 | 基于历史记录进行搜索、提问和追问 |
| Summary | 总结 | AI 基于记录生成的内容概括 |

它们之间的关系是：

> 先保存记录，再查看历史，最后可以根据历史记录提问或生成总结。

「历史」只用于描述过去内容这一领域概念。Web 导航与页面标题使用「全部记录」，避免和会话历史混淆；问答会话区统一使用「问答」，归档视图明确显示「已归档问答」。

## 记录

记录是 Sillage 的唯一内容单位。一天可以有多条记录，每条可以很短，也可以是一大段。

记录适合保存：

- 一件发生的事
- 一种情绪
- 一个想法
- 一张照片
- 一段关系里的细节
- 一个暂时还没整理清楚的内容

记录不强制标题。用户只需要写日期和正文即可。

## 总结

总结是 AI 基于记录生成的内容概括。

总结不是诊断，也不是对用户下结论。它应该帮助用户了解：

- 今天或本周主要发生了什么
- 哪些主题反复出现
- 哪些内容值得回看
- 事件、状态或表达方式有什么变化

每个总结都必须说明它基于哪些记录生成，并提供回到原文的入口。

示例：

> 基于 8 条记录生成。

## 问答

问答是基于历史记录的对话功能。

它包括：

- 关键词搜索
- 根据记录提问
- 会话历史
- 分支追问
- 回答中的来源引用

问答功能的目标是帮助用户从历史记录中找到信息、总结内容、继续追问。

## 信息架构

主导航：

- 写记录
- 全部记录

问答入口保留在左侧栏的「开始问答」按钮与「问答」会话区中，不再作为主导航项重复出现。会话区提供服务端搜索和已归档视图，不为这些管理动作增加主导航项。

页面职责：

| 页面 | 作用 | 承载的概念 |
| --- | --- | --- |
| 写记录 | 主要写作入口：统一编辑器 + 今日记录 + 最近记录 | 记录 |
| 全部记录 | 按时间查看所有记录，支持列表与日历视图 | 历史 / 记录 |
| 问答 | 搜索、根据记录提问、追问、回看会话 | 问答 |

全局：除「问答」对话页外，任意页面都有「速记」入口（右下角悬浮按钮 / ⌘·Ctrl+J），用于快速保存记录。速记正文作为全局草稿保留，进入问答页只隐藏入口，重载后仍可恢复；非空草稿同时保护关闭、刷新和手动退出。保存进行中锁定输入、关闭和跳转入口，重复点击或快捷键只产生一次保存请求；只有保存成功才清除草稿，失败时保留内容供重试。

设置放在左侧导航栏底部用户卡片的展开菜单中，不作为主导航项。旧路径 `/review` 会重定向到 `/ask`，`/ask` 是问答的规范入口。

主题切换在左侧栏底部提供快捷按钮（移动 Web 位于导航抽屉内），并在「设置 → 外观」保留带文字的完整入口；两处操作同一份主题偏好，任一入口切换后，两处图标与文案立即同步。

Web 的 AI 设置表单以最近一次成功读取或保存的内容为基准；存在未保存修改时，站内跳转和手动退出登录先确认，关闭或刷新页面也会触发离站保护。首次读取失败时只显示重试状态，不能让用户进入没有可保存基线的空表单；跨设置分类切换后，未保存提示与保存入口仍保持可见。保存或删除请求进行中锁定可修改表单，保存失败后仍保持未保存状态；删除已有档案前必须二次确认，且存在其他未保存修改时应先保存，不得用删除请求覆盖当前表单。

## 写记录页

写记录页是首页，也是主要写作入口。用户打开后应清楚知道可以开始记录。

页面结构：

- 顶部显示日期和标题「今天想记录什么？」
- 中央为统一编辑器
- 下方展示今天的记录
- 下方展示最近记录

主占位文案：

> 写下想记录的内容…

编辑器只保留日期与 Markdown 正文。记录、速记、详情、时间线、日历等当前界面不再保留天气、地点、标题、预设心情、人物、标签、关系等显性字段。

AI 默认不打扰写作。用户可以主动生成总结，也可以在设置里开启保存后自动生成。

Web 的新建与编辑器按「新记录 / 单条记录」分别保存未提交草稿。页面重新载入后恢复对应草稿；有改动时，站内跳转先显示「继续编辑 / 离开此页」确认，关闭或刷新页面也会触发离站保护，保存成功后清除草稿。编辑页点「取消」时还需明确选择继续编辑或放弃修改；放弃会删除对应草稿。草稿保护不改变记录仍需用户明确保存的原则。

编辑已有记录时，草稿同时记住它所基于的服务器版本。如果服务器记录已在其他位置更新，页面先显示服务器最新内容，不自动用旧草稿覆盖；用户明确选择「保留最新内容」或「恢复我的草稿」前不得提交更新，恢复后仍需检查并再次保存。

详情页在最新记录请求返回前不允许进入编辑或执行版本动作；编辑正文与提交版本必须来自同一个详情快照。只有服务端明确返回 404 才显示记录不存在；临时网络或服务端错误应保留已缓存正文并允许重试。从首页、全部记录页筛选、日历或问答来源打开详情后，返回应回到原来的路径与筛选状态。

附件上传完成前不允许保存或取消编辑；站内跳转会说明附件仍在上传并要求明确选择。这样可避免先保存缺少附件 Markdown 的正文，也避免普通误触直接丢弃上传中的修改。

## 全部记录页

全部记录页用于按时间查看所有记录。

全部记录页应支持：

- 列表视图
- 日历视图
- 那年今日
- 在列表视图切换「未归档 / 已归档 / 收藏」，并能搜索、打开、收藏、取消收藏和取消归档记录；列表与搜索都由服务端按所选状态过滤，不只检查当前已加载的列表
- 打开记录详情
- 在记录详情中手动生成总结
- 三个状态视图互斥：收藏记录只在收藏视图显示；取消收藏后按归档状态回到未归档或已归档视图
- 日历在完整历史加载完成后才显示结果；加载过程显示已读取数量，失败后停止并提供手动重试
- 分页加载同一游标时只允许一个进行中请求；刷新或记录变更后的缓存不得被旧列表、旧分页响应覆盖

全部记录页不显示记录种类标签，也不提供按记录种类筛选。

## 问答页

问答页是独立主入口，用于搜索、提问、追问和历史回看。

它应承载：

- 根据记录提问
- 会话历史与分支追问
- 搜索完整的问答标题与消息正文，而不是只过滤侧栏已加载的会话
- 归档、查看已归档问答和恢复问答；归档不是删除
- 可按本轮需要切换的内容来源：记录、单条总结、总结
- 空状态里的搜索入口
- 回答中的来源引用与回到原文入口

入口文案：

> 根据记录提问

用户可以问：

- 最近我反复提到什么？
- 这一周有哪些重要内容？
- 我最近的状态有什么变化？
- 我提到某个人时，情绪有什么变化？
- 哪些事情反复出现了？

回答应包含：

1. 简短回答
2. 相关来源
3. 重点内容
4. 可继续追问的方向

同一时间只提交一个问题或重新生成请求。回答生成期间再次按 Enter 或点击发送不会启动第二条流；此时继续输入的文字保留在编辑框中，可在当前回答结束后再提交。

创建新会话期间如果用户切换会话或再次点击「开始问答」，晚到的旧创建结果不得切回旧会话、启动回答或把旧问题写入当前输入框。

Web 默认会话列表只显示未归档问答。会话搜索和已归档视图由服务端先按归档状态过滤再应用数量限制；搜索或视图切换时，迟到响应不得覆盖当前结果。归档与恢复无需二次确认，但必须可逆并提供结果反馈；归档当前会话后回到空白问答。回答生成期间禁止切换或归档会话。

## AI 行为准则

AI 在 Sillage 中是辅助功能，不是用户表达的替代品。

AI 可以做：

- 生成总结
- 识别情绪
- 发现反复出现的主题
- 提出追问
- 回答记录相关问题
- 把问答回答保存为记录

AI 不应该做：

- 替用户下心理判断
- 使用诊断式语言
- 把猜测包装成事实
- 夸大没有证据的关联
- 在用户写作时强行打断
- 让 AI 产物比原始记录更重要

AI 语气应当：

- 清楚
- 直接
- 具体
- 有依据
- 说明不确定性
- 尊重用户最终解释权

避免：

> 你其实很缺爱。

推荐：

> 在这几条记录里，你多次写到“没关系”和“算了”，这些内容都和等待、失落、没有被回应有关。可以回看这些记录，确认它们是否描述了同一类情况。

推荐句式：

- 这些记录里反复出现了……
- 这几条记录都提到了……
- 目前能看到的依据是……
- 记录中没有足够信息确认这一点。
- 如果要继续看，可以追问……

- 查看总结

## 已确定决策

- 产品名使用 `Sillage`。
- 首次访问创建唯一账号；初始化完成后默认登录使用，不提供免登录公开试用入口。
- Web 主导航使用「写记录 / 全部记录」；问答通过「开始问答」与「问答」会话区进入，会话搜索和归档不增加主导航项。
- 产品只保留单一内容单位「记录」，不再区分记录种类。
- 「洞察」统一改为「总结」。
- AI 汇总基于记录和总结，不再按记录种类选择来源。
- 首页采用统一编辑器，不强迫用户一开始选择类型。
- AI 默认可切换、可主动召唤，不在写作时强行打扰。
- AI 总结必须可追溯到原始记录。
- 产品语言应通俗、明确、具体。
- 不保留单独的「照见」一级入口；问答页不再提供「整理记录」入口。
- 记录界面只保留日期与正文，不保留天气、地点、标题、预设心情、人物、标签、关系等显性字段。
- 设置放进左侧用户卡片的展开菜单，不作为主导航项。
- 问答页采用全局左侧栏承载最近会话，最近会话默认常显，页面主体保持单列聊天体验。
- 原生 Android 使用 Kotlin + Jetpack Compose，不做 WebView 套壳；当前已覆盖在线 / 离线记录、附件、AI 总结、Ask、导入导出和手动同步，仍不做后台自动同步或推送通知。
- Web 编辑器按记录隔离并恢复未保存草稿；Android 编辑器在正文或日期变更后拦截返回并确认是否放弃。
- Web 与 Android 全部记录页提供「未归档 / 已归档 / 收藏」分段入口；三个视图互斥，归档不是删除，归档记录仍可搜索、打开和取消归档。
- Web 问答会话支持服务端搜索及归档 / 恢复；默认列表不显示已归档会话，Android 默认列表同样过滤同步得到的归档会话。

## 后续可细化事项
